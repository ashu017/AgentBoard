# CEO Plan: AgentBoard — Agent Status Wall (SUPERSEDED)

Generated by /plan-ceo-review on 2026-06-26
Branch: unknown (not a git repo; design originated in a brainstorming conversation) | Mode: SELECTIVE EXPANSION
Repo: none yet (greenfield, intended to be open-sourced, MIT)

## Problem

Running multiple AI agents concurrently, there is no glanceable way to see what each
agent is doing right now — which are working, which are stuck, which have died. Today
the only option is tailing N terminal windows / log files, which gets linearly worse
with agent count. Pain is real at ~3+ concurrent agents.

This is fundamentally an **agent observability / liveness** problem presented as a
Kanban board. The Kanban metaphor (work items flowing Todo→Done) and the stated
outcome ("what is each agent doing right now") are two lenses on the same data; the
design must serve both.

## Landscape (in-distribution knowledge; live search was unavailable)

- **Layer 1 (tried-and-true):** Generic PM boards (Trello, Linear, JIRA, GitHub
  Projects) + their REST APIs. Human-paced, heavyweight, no agent-liveness concept.
- **Layer 2 (new/popular):** Agent observability platforms — LangSmith, Langfuse,
  AgentOps, Arize Phoenix, Helicone. These are trace/eval tools (LLM spans, tokens,
  latency), not glanceable "what is each agent doing right now" status walls.
- **Layer 3 (first principles):** A lightweight, self-hostable **agent status wall**
  is a distinct product from both. The wedge: glanceability + zero-friction HTTP push
  + clone-and-run self-hosting. Underserved.

## Chosen Approach (D2)

**Approach A: Single Next.js (App Router, TypeScript) app + SQLite (better-sqlite3) +
in-process SSE.** One repo, one process, one language. Clone-and-run is the #1 OSS
adoption lever. The only real limitation (in-process EventEmitter → single-instance
SSE) does not matter for a status dashboard; Postgres + LISTEN/NOTIFY is documented as
the scale escape hatch.

Alternatives considered and rejected:
- **B (Fastify API + Vite React SPA):** lighter runtime but two build artifacts /
  more contributor setup; contradicts the stated Next.js preference.
- **C (Next.js + Postgres):** scalable realtime but requires standing up a database
  before the app runs — destroys clone-and-run; premature for v1.

## Vision

### 10x Check
The default open-source "agent status wall" — a drop-in any agent stack POSTs to, that
answers at a glance: which agents are alive, what each is doing right now, what's
stuck or dead, and what happened over the last hour. Liveness + current-task +
history + (later) alerting when an agent stalls or dies.

## Core Scope (baseline, from brainstorming)

- Single Next.js + TypeScript + SQLite app; clone-and-run.
- HTTP push ingest: `POST /api/tasks`, `PATCH /api/tasks/:id`.
- Reads: `GET /api/tasks` (board load), `GET /api/tasks/:id` (card + timeline).
- Live updates via SSE: `GET /api/stream`, bridged from an in-process EventEmitter.
- Task-centric Kanban, five statuses: Todo / In Progress / Blocked / Failed / Done.
- `tasks` table + `task_events` history table (every write appends an event).
- Optional shared-token auth: if `AGENTBOARD_TOKEN` is set, writes require
  `Authorization: Bearer <token>`; reads/UI open. If unset, fully open (zero-config).
- Read-only React UI (no human editing; the board reflects what agents report).
- MIT license, README quickstart, .env.example, CONTRIBUTING.md.

### Data Model (explicit)

`tasks`:
- `id` TEXT PK — **client-supplied stable ID** (so an agent can PATCH/heartbeat
  without a create round-trip first; `POST` upserts on conflict). Server falls back
  to a generated UUID if the client omits it on create.
- `agent_id` TEXT NOT NULL — the reporting agent's self-chosen identifier. Load-bearing
  for the roster view (D4) and agent-level liveness (D3).
- `title` TEXT NOT NULL
- `status` TEXT NOT NULL — canonical lowercase wire values:
  `todo` / `in_progress` / `blocked` / `failed` / `done` (display labels "Todo", "In
  Progress", etc. are a UI concern). `todo`/`in_progress`/`blocked` are *active*;
  `done`/`failed` are *terminal*. **No transition validation** — any of the five
  accepted on any write (agents are the source of truth; the board does not police
  their lifecycle). Unknown values rejected 400.
- `detail` TEXT NULL — current free-text activity.
- `project` TEXT NULL — grouping/filter key (D6).
- `metadata` TEXT NULL — JSON blob, validated as parseable JSON on write, **capped at
  4 KB** to prevent agent-side footguns (D6). Over-cap → 400.
- `created_at`, `updated_at` INTEGER (epoch ms). `updated_at` bumps on every write.

`task_events`:
- `id` TEXT PK — **server-generated UUID per event**. `task_id` TEXT FK → tasks(id), `event_type` TEXT
  (`created` / `status_changed` / `detail_updated`), `from_status` TEXT NULL,
  `to_status` TEXT NULL, `detail_snapshot` TEXT NULL, `created_at` INTEGER.
- **One event per write.** `event_type` precedence: a brand-new id → `created`; an
  upsert/PATCH that changes status → `status_changed`; otherwise → `detail_updated`.
  A single write that changes both status and detail emits one `status_changed` row
  (the `detail_snapshot` captures the new detail). No multi-event writes.

`agents` (**persisted projection table**, the single source of truth for liveness — not
a runtime GROUP BY):
- `agent_id` TEXT PK, `last_seen_at` INTEGER (epoch ms), `first_seen_at` INTEGER.
- Upserted on **every** write that carries an `agent_id` (POST, PATCH) and on every
  heartbeat: `last_seen_at = max(existing, now)`. Because it is persisted and updated
  on heartbeat, it **survives task pruning** — an agent with only pruned terminal tasks
  still appears in the roster as long as it keeps heartbeating. The roster joins
  `agents` (for `last_seen_at`/staleness) to `tasks` (for the current task), so the
  earlier "derived by grouping tasks" phrasing is superseded by this explicit table.

### Error contract

- `400` — invalid body (zod): unknown status, unparseable/over-cap metadata, missing
  required field. Body: `{ error: "<message>" }`.
- `401` — token required and missing/wrong. Body: `{ error: "unauthorized" }`. Applies
  to **all** write endpoints including `POST /api/agents/:agent_id/heartbeat`.
- `404` — PATCH to unknown task id. Body: `{ error: "not found" }`.
- **PATCH = partial update:** only fields present in the body change; omitted fields
  (`detail`, `project`, `metadata`, `status`) are left untouched. To clear a nullable
  field, send explicit `null`. **POST upsert on an existing id** merges the same way
  (provided fields overwrite, omitted fields preserved), so a re-POST never silently
  nulls prior `metadata`/`project`.
- Concurrent PATCHes: last-write-wins on the `tasks` row (acceptable for a status
  wall); each still appends its own `task_events` row, so history is not lost. Each
  row+event write is wrapped in a **single better-sqlite3 transaction** so a crash can
  never leave a task row without its event (or vice versa).
- **`POST /api/agents/:agent_id/heartbeat`:** empty body; success `200 { last_seen_at }`.
  **Auto-creates** the `agents` row if the agent_id is new (heartbeat-before-any-task is
  valid — it just registers a live agent with no current task yet). Never 404s.
- Tiebreaker for "current task" when two tasks share `updated_at`: higher `id` wins
  (stable, deterministic).
- **Read limits:** `GET /api/tasks` caps the board at the most-recent N active +
  recently-terminal tasks (default 500, configurable); `GET /api/tasks/:id` timeline
  paginates `task_events` (default newest 100). Prevents unbounded read payloads when
  retention is disabled.

## Accepted Scope (added to this plan)

- **Liveness (agent-level is the source of truth):** the persisted `agents` table
  (defined above) keyed by `agent_id` holds `last_seen_at`, upserted with
  `last_seen_at = max(existing, now)`. Any write (POST/PATCH) freshens it; a dedicated
  `POST /api/agents/:agent_id/heartbeat` lets an agent signal "still alive" mid-task
  without a status change. **Task-card staleness is derived from the owning agent's
  `last_seen_at`, not a separate per-task clock** — this resolves the per-task vs
  per-agent ambiguity: there is one liveness clock per agent, and both the card and the
  roster read it. Configurable thresholds via env: `STALE_WARN_SECONDS` (→ amber),
  `STALE_DOWN_SECONDS` (→ red). Thresholds are exposed to the browser in the initial
  `GET /api/tasks` payload (and on SSE connect) so the client can compute color/relative
  time locally without round-trips. **Staleness coloring applies only to cards in
  *active* statuses (todo/in_progress/blocked).** Terminal cards (done/failed) are never
  stale-colored — a finished task shouldn't turn red just because its agent later went
  quiet. The roster still reports the agent's true `last_seen_at`/staleness independently.
- **Agent-roster view:** toggle from the Kanban; one row per agent with current task,
  status, `last_seen_at`, and staleness. Built by reading the persisted `agents` table
  (for `last_seen_at`/staleness) and joining to `tasks` for each agent's current task
  (most recently updated active task, else most recent task, `id` as tiebreaker).
  Because `agents.last_seen_at` is persisted and refreshed on heartbeat, pruning a
  terminal task never makes a live agent vanish from the roster. Live over the same SSE
  stream.
- **Reporting snippets:** copy-paste curl + fetch examples in the README for create,
  update, and heartbeat. No published package in v1.
- **Grouping/metadata:** nullable `project` (string) + nullable `metadata` (JSON
  text) columns on `tasks`; a project/tag filter control on both views.
- **Retention:** `RETENTION_DAYS` env, **default unset = pruning disabled** (explicit
  opt-in; zero surprise data loss). When set, a `setInterval` timer in the long-lived
  Node process (hourly) runs a guarded delete: removes only terminal (done/failed)
  tasks whose `updated_at` is older than the cutoff, plus their `task_events`. Never
  deletes active (todo/in_progress/blocked) tasks. Agent `last_seen_at` is derived from
  a lightweight `agents` projection that is updated on every write, so it survives task
  pruning — a long-idle but still-heartbeating agent stays visible in the roster.

## Runtime & Reliability Constraints (resolved from spec review)

- **SSE reconnect (subscribe-then-refetch, no missed events):** on (re)connect the
  client opens the `EventSource` **first**, buffers any events that arrive, then issues
  `GET /api/tasks`; once the snapshot is applied it drains the buffer. Combined with
  last-write-wins by `updated_at`, a replayed/duplicated event is idempotent, so the
  board converges with no drift after laptop sleep / proxy timeout. No `Last-Event-ID`
  needed. Server sends an SSE heartbeat comment (`:\n\n`) every ~30s.
- **Next.js singleton story (the real implementation risk):** the SQLite connection,
  the SSE `EventEmitter`, and the retention timer must each be **one instance** across
  route-module evaluations and dev HMR. Pin them on `globalThis` (e.g.
  `globalThis.__agentboard_db ??= openDb()`) and start the retention `setInterval` +
  emitter from **`instrumentation.ts` `register()`** (Next's documented once-per-process
  server hook), not from a route module. This is called out so the implementer doesn't
  rediscover it the hard way.
- **Deployment runtime constraint:** in-process EventEmitter SSE requires AgentBoard to
  run as **a single long-lived Node process (`next start`), not a serverless/edge
  deploy** (Vercel functions don't hold a long-lived emitter). Documented in the README
  as the supported deploy model; Postgres + LISTEN/NOTIFY is the multi-instance escape
  hatch (deferred).
- **better-sqlite3 native binding:** README notes a supported Node version range;
  native rebuilds on Node-version mismatch are the most likely clone-and-run friction.
- **Retention timer:** an hourly `setInterval` started from `instrumentation.ts` (not
  startup-only), so a 24/7 process actually bounds growth.
