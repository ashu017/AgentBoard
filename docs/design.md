# CEO Plan: AgentBoard — Agent Manager Console (Platform v2)

Generated by /plan-ceo-review on 2026-06-26
Branch: unknown (not a git repo; design from conversation) | Mode: SCOPE REDUCTION
Repo: none yet (greenfield; deploy target Vercel)

## Problem

A person managing multiple AI agents has no console to **assign work to a fleet and
watch it get done**. Today they prompt each agent by hand. They want to log in, onboard
their agents, hand out tasks like a manager, and see a task board with live status —
while the agents themselves read and update their tasks programmatically (via MCP).

This is an **active coordinator with a human in the loop** — a deliberate reversal of
the prior passive-observatory design (see superseded plan
`2026-06-26-agentboard.md`). The manager is the source of work; agents are the
workforce; the board is the shared truth.

## Landscape (in-distribution knowledge; live search unavailable)

- **Layer 1:** Human task managers (Linear/JIRA/Asana) — no agent-as-assignee concept.
- **Layer 2:** Agent orchestration frameworks (CrewAI, AutoGen, LangGraph) assign tasks
  to agents *in code*, no manager-facing board with a human in the loop. MCP is the
  emerging standard for how agents consume external tools.
- **Layer 3 (wedge):** A human-in-the-loop manager console that bridges a task-assignment
  UI to an MCP agent plane. Underserved; stronger product than the observatory.

## Chosen Stack (D9)

**Next.js on Vercel + Supabase (Postgres + Auth + Realtime + Row-Level Security).**
- **Why:** multi-tenant isolation is the highest-stakes correctness property here (a
  wrong query leaks one manager's tasks and agent keys to another). RLS enforces it **at
  the database**, not in app code. Supabase bundles the DB, human auth, and the realtime
  the live board needs, and self-hosts (keeps an OSS path open).
- **MCP server** built with the **official MCP TypeScript SDK** in its **stateless
  Streamable-HTTP transport** mode, mounted as Next.js route handlers. Stateless mode
  (no long-lived session held server-side) is what fits Vercel function limits; the 3 MVP
  tools are all short request/response. Using the real SDK (not a bespoke JSON endpoint
  labeled "MCP") is what guarantees standard MCP clients — the agents users actually run —
  connect out of the box. **Spike first** (before committing the rest of the agent plane):
  stand up the SDK's stateless transport on a Vercel route handler and connect a real MCP
  client end-to-end, to confirm the handshake/JSON-RPC framing works within serverless
  execution limits. **The spike must connect a REAL MCP client** (not just curl JSON) and
  verify the client doesn't expect a held-open `text/event-stream` the serverless function
  can't sustain. **Pin the MCP SDK version** (transport behavior has been a moving target).
  **Named fallback if the spike fails (Eng-review D9):** host the MCP route on a
  non-serverless target (Fly.io / Railway / a long-running container) while the rest of the
  app stays on Vercel — so a failed spike doesn't strand the agent plane. Agents
  authenticate per-request with their API key (bearer).
- Rejected: **B (Neon + Auth.js + Pusher)** — assembles/secures auth+tenancy+realtime
  separately, most surface to leak. **C (Convex + Clerk)** — best realtime but two
  vendors, less-conventional model, tenancy still in function code.

## Vision (12-month)

The default "agent manager" console: assign work to a fleet, agents pull via MCP, live
board, results review, approvals, multi-user workspaces, audit trail.

## Decision log (referenced as D# below)

- **D9** — Stack: Next.js/Vercel + Supabase (Postgres+Auth+Realtime+RLS). Chose RLS-
  enforced tenancy over roll-your-own (Neon+Auth.js+Pusher) or Convex+Clerk.
- **D11** — Task handoff: directed assignment (manager assigns each task to one specific
  agent), not a pull/claim pool. Avoids claim-race concurrency in the MVP.
- **D12** — Agent auth: per-agent long-lived API key, shown once, hashed at rest,
  revocable. Short-lived/rotating tokens deferred.
- **D13** — Agent-plane isolation. **Resolved D15: DEFERRED.** v1 uses an app-code
  scoped query for the agent path; the RLS/GUC mechanism is deferred to the multi-user
  milestone (see below). The detailed RLS design is retained in an appendix for that
  milestone.
- **D15** — Cross-model decision (outside voice vs inside review): defer the RLS/GUC
  agent-isolation mechanism. Rationale: v1 is single-tenant (one user owns everything),
  so the cross-tenant threat does not exist until multi-user ships — which is already
  deferred. Defer the defense to match the threat. The mandatory cross-tenant isolation
  test stays in v1 so the app-code predicate cannot silently go missing, which also makes
  the later RLS retrofit a bounded, test-guarded migration rather than a rewrite.

## MVP — the one loop that must work (SCOPE REDUCTION)

**Loop:** manager logs in → creates a task → assigns it to a specific agent → agent
reads it via MCP → agent updates status / submits result via MCP → manager sees the
change live on the board.

This loop is the whole risk. Everything not required to prove it is a follow-up.

### In scope (irreducible — the loop breaks without each)

| Component | Minimum |
|---|---|
| Human login | Supabase Auth, **GitHub OAuth** (single provider for v1; the agent audience is developer-shaped, and OAuth avoids email-deliverability setup). |
| Tenancy | **Single workspace per user; RLS on every table.** Isolation cannot be retrofitted safely, so it's in from line 1. |
| Agent entity | Create agent (name, optional description); one **API key** per agent: shown once, hashed at rest, revocable (D12). |
| Task | Create + assign to one agent + status. Statuses cut to **todo / in_progress / done / failed** (D11: directed assignment, one assignee per task). |
| MCP server | **3 tools:** `list_my_tasks(status?)`, `update_task_status(task_id, status, note?)`, `submit_result(task_id, output, status?)`. Authenticated by agent API key; scoped to that agent's own tasks (mechanism = D13). |
| Board | Manager sees their tasks grouped by status; updates **live via Supabase Realtime** subscribed to **`tasks` row changes** (insert = new card, update = status move / result arrival). `task_events` is history, not the live feed. |

> **Honest scope note:** `task_events` is **not** strictly required to prove the core
> loop (the loop runs on `tasks` rows alone). It's included as a cheap, deliberate
> exception because the RLS + write-path pattern is identical to `tasks`, and an
> append-only audit trail is far cheaper to add now than to reconstruct later. Everything
> else in this table genuinely is irreducible.

### MCP tool semantics & task state machine

- **Single source of truth (DRY):** the status enum and the legal-transition map live in
  **one shared TS module** (e.g. `lib/task-status.ts`: `STATUSES`, `isTerminal()`,
  `canTransition(from, to)`). The MCP validator, the agent-db layer, the board UI columns,
  and the tests all import it. The DB `CHECK` constraint mirrors the enum and is kept
  honest by a test that asserts the constraint matches `STATUSES`. Adding/changing a
  status is then a one-place edit. No per-layer copies.
- Allowed status values: `todo`, `in_progress`, `done`, `failed`. Manager creates tasks
  as `todo` (assigned). Agent transitions: `todo → in_progress`, `todo → failed`
  (reject before starting), `in_progress → {done | failed}`, and `in_progress → todo`
  (decline/reset). `done`/`failed` are terminal; moving out of terminal → `409`. Invalid
  status value → `400`.
- `update_task_status` / `submit_result` on a `task_id` **not assigned to the calling
  agent** → `404` (never `403`), so the agent plane never reveals another agent's tasks.
  (In v1 this is enforced by the mandatory `WHERE assigned_agent_id = $resolvedAgent`
  scoping in the agent-db module: a non-matching row simply isn't returned, which the
  handler maps to 404. At the multi-user milestone, Appendix-A RLS makes the row invisible
  at the database for the same effect.)
- `submit_result(task_id, output, status?)` sets `result` only on an **`in_progress`**
  task (an agent submitting a result implies it started work; calling it on a `todo` task
  → `409`). If
  `status` is supplied (must be terminal: `done`/`failed`), it submits the result **and**
  transitions in one transaction; if omitted, status is unchanged and the agent may call
  `update_task_status` separately. `output` is capped at **256 KB**; oversize → `413`
  (protects Realtime payload size and board render).
- **Error contract (MCP + REST):** `400` invalid input / unknown status, `401` missing or
  revoked/unknown API key, `404` task not assigned to caller (or absent), `409` illegal
  transition (e.g. out of terminal), `413` result over cap.
- Every successful MCP call refreshes the agent's `last_seen_at` — but **throttled to at
  most once per 30–60s per agent** (skip the write if the stored value is already within
  the window). Without this, `list_my_tasks` (the hottest path — agents poll it in a loop)
  turns every read into a DB write **and**, because `agents` is under Realtime for the
  connected indicator, a Realtime event — write amplification + event flooding for a
  liveness dot. 30–60s granularity is plenty for "connected / last seen" (Eng-review D10).
  Heartbeat tool still deferred.
- **Abuse/rate-limiting (known, deferred):** the MCP endpoint is a public,
  bearer-authenticated surface on the internet. v1 ships without rate-limiting but this is
  a *known* gap, flagged here and in TODOs — not an oversight. A per-key rate limit
  (Upstash/Vercel KV or Supabase) is the first follow-up if abuse appears.

### Data model (MVP)

`users` — Supabase Auth managed.
`workspaces` — `id`, `owner_user_id` (**UNIQUE** — DB-enforces one workspace per user),
  `name`, `created_at`.
  **Bootstrap (revised per Eng-review D7, reversing the earlier trigger choice):** an
  app-code `getOrCreateWorkspace(userId)` runs in the **shared authenticated-session
  layer** (so it fires on the first authenticated request regardless of entry path),
  doing an idempotent upsert. The `UNIQUE(owner_user_id)` constraint guarantees exactly
  one workspace even under a race. This keeps signup robust (a workspace bug can't break
  GitHub OAuth signup the way an `auth.users` trigger would) and the logic plain, testable
  TypeScript — no `SECURITY DEFINER` / `search_path` auth-schema footgun.
`agents` — `id`, `workspace_id`, `name`, `description?`, `api_key_hash`,
  `api_key_prefix` (first ~8 chars, for UI display/identification only), `revoked_at?`
  (NULL = active; set = revoked — single disable mechanism, no separate `status`),
  `last_seen_at?` (updated on every successful MCP write), `created_at`.
`tasks` — `id`, `workspace_id`, `assigned_agent_id` (NOT NULL in MVP — directed),
  `title`, `description?`, `status` (todo/in_progress/done/failed), `result?` (text),
  `created_by_user_id`, `created_at`, `updated_at`. **Indexes:**
  `tasks(workspace_id, assigned_agent_id)` for `list_my_tasks` (every agent poll) and
  `tasks(workspace_id, updated_at DESC)` for board load + the read cap. Keeps both hot
  read paths off table scans as tasks accumulate in long-running deployments.
`task_events` — `id`, `task_id`, `actor_type` (user/agent), `actor_id`, `event_type`
  (created/status_changed/result_submitted), `from_status?`, `to_status?`, `note?`,
  `created_at`. Append-only history. Index `task_events(task_id, created_at)` for the
  card-drawer timeline. (No separate `assigned` event — tasks are created
  already assigned in the MVP, so `created` covers it; reassignment is out of scope.)
  **Both planes write events** (human: `created`; agent: `status_changed`,
  `result_submitted`), so the row is constructed by a **single `appendTaskEvent()` helper**
  (not hand-built in each path) to keep field population consistent. Each task-row write +
  its event append run in **one transaction** so history can't desync from state.

**API key:** generated as `ab_<prefix>_<secret>` where the **`<secret>` segment alone
carries ≥256 bits** of entropy (the `<prefix>` is public/displayed, so it must not be
counted toward secret strength). Stored as a **plain SHA-256 of the full token** in an
indexed `api_key_hash` column — no KDF (bcrypt/argon2) is needed because the token's
≥256-bit entropy already makes brute force infeasible, and a per-row-salted KDF would
prevent the `WHERE api_key_hash = ?` indexed lookup the MCP path needs on every call. Shown to the manager exactly once at creation;
only the prefix is ever displayed afterward. Revoke = set `revoked_at`; revoked keys fail
the lookup immediately.

**Isolation posture (v1, per D15):**
- **Human plane — RLS now.** Every table scoped by `workspace_id`; a row is visible if
  its workspace's `owner_user_id = auth.uid()` (standard Supabase user-JWT RLS). This is
  free with the stack, so it stays in v1.
- **Agent plane — app-code scoped query now, RLS deferred.** Per MCP request: SHA-256 the
  bearer key, look up the `agents` row by `api_key_hash` where `revoked_at IS NULL`,
  resolve `(agent_id, workspace_id)`, then scope **every** task/event query with
  `WHERE workspace_id = $resolvedWs AND assigned_agent_id = $resolvedAgent`. In v1 there
  is one tenant, so there is nothing cross-tenant to leak; the predicate is correctness
  hygiene, not a security boundary yet. The **mandatory cross-tenant isolation test**
  (below) runs against this path so the predicate can never silently go missing — which
  also makes the later RLS retrofit a bounded, test-guarded change.
- **Agent-path DB role (v1):** because the agent request carries no Supabase user JWT,
  it **cannot** use the `authenticated`/`anon` role — human-plane RLS (`owner_user_id =
  auth.uid()`) would hide every row and queries would return empty. So the agent path
  connects via the **service-role** (RLS-bypassing) key from a **single isolated,
  server-only module** (`lib/agent-db.ts` or similar) that exposes only the scoped
  queries. Service-role never touches the human UI path. (At the multi-user milestone,
  this module is what gets replaced by the Appendix-A RLS mechanism.)
- **Structural scope enforcement (Eng-review D8, strengthening the test-only floor):**
  since service-role removes the DB-level safety net, app-code scoping is the *only*
  agent-plane boundary in v1 — so it must be enforced structurally, not by convention.
  Every query in `agent-db.ts` goes through a **scoped-query wrapper that takes
  `(workspace_id, agent_id)` as required arguments** and injects the `WHERE` itself;
  there is no code path to issue a raw unscoped query from this module. A future query
  added without scope won't typecheck/run, rather than silently shipping green past the
  cross-tenant test (which only covers paths it exercises). The cross-tenant test remains
  as a second layer.
- **Deferred to the multi-user milestone (when a second tenant first exists):** the full
  DB-enforced agent-plane RLS via SECURITY DEFINER bootstrap lookup + transaction-local
  GUC + transaction-mode pooling. Design retained in Appendix A so it's ready to drop in.

> The full DB-enforced agent-plane RLS mechanism (SECURITY DEFINER bootstrap + GUC +
> transaction-mode pooling) is **deferred to the multi-user milestone** per D15 and lives
> in **Appendix A** at the end of this doc, ready to drop in when a second tenant exists.

**Live board / Realtime — sequenced de-risking gate (Eng-review D9), not a config aside:**
the board is human-plane (user JWT) and Supabase Realtime re-applies RLS before delivering
a Postgres Change to a subscribed client. **Critical dependency made explicit:** the agent
writes its row via the **service-role** path, but the *manager's* board only receives the
live update if that agent-written row **passes the human-plane RLS SELECT policy** for the
manager. If that policy is even slightly wrong, the write commits to the DB, the board
**never updates, and nothing errors** — a silent failure of the single most-demoed feature.
- **Prove-first gate:** before building board features, write an isolation test that
  performs an agent-plane (service-role) write and asserts a subscribed board client
  **receives** the Realtime event under RLS. This makes the silent failure loud.
- Requires the realtime authorization setup (replication on `tasks` + realtime-schema
  policies in current Supabase versions) — verified by that gate, not assumed.

### Auth: two planes (kept separate)

- **Human:** Supabase Auth session (one provider). Drives the board UI and management.
- **Agent:** per-agent API key (bearer) presented to the MCP endpoint, verified by hash
  lookup, mapped to agent + workspace. Revoking the key disables the agent's access
  immediately. No human-session capabilities.

## Manager UI (the human-facing half)

Three screens, all human-plane (Supabase session). Read-from-Realtime, write-via-server-
actions/route-handlers. Deliberately minimal — this is the surface that proves the loop,
not a polished product.

**First-run journey (the arc the UI must support):** GitHub login → empty board with a
single "Add your first agent" CTA → create agent → **shown-once key + ready-to-paste MCP
snippet** → manager wires the key into their agent externally → **Agents screen flips that
agent to "connected"** the moment its key is first used (closes the "did it work?" gap) →
create + assign a task → **the card moves on its own when the agent reports** (the magic
moment that proves the product). The two previously-unmarked beats — connection
confirmation and the first live card move — are now first-class: the connected indicator
(Screen 2) and the live board (Screen 1) respectively.

### Screen 1 — Board (the payoff, default route after login)

- **Information hierarchy (what the manager sees 1st / 2nd / 3rd):** the board is a live
  monitor, so it's optimized for the 3-second scan "is anything wrong?":
  1. **Scan summary line** (top, above columns): `⚠ 1 failed · 3 in progress · 2 done`.
     The failed count is the loudest token; zero-failed shows a calm "all healthy".
  2. **In Progress + Failed columns** carry the most visual weight — what's active and
     what broke. **Failed** gets the loudest treatment (red rail/badge); a non-empty
     Failed column should be impossible to miss.
  3. **Done** is the quietest column — muted, and collapsible/scrollable so it doesn't
     dominate as it accumulates over a long run. Todo sits between.
  A healthy board must still read as calm (no alarm styling when failed count is 0).
- **Layout:** four columns — Todo / In Progress / Done / Failed, weighted per the
  hierarchy above. Each task is a card showing title, assigned agent (name + monospace
  key-prefix), a relative `updated_at`, and a truncated `result` preview on terminal
  cards (error excerpt on Failed). Click a card → detail drawer with full `description`,
  full `result`, and the `task_events` history timeline.
- **Live:** subscribed to `tasks` row changes via Supabase Realtime. Insert = card
  appears; update = card moves column / result fills in. No manual refresh.
- **Entry point:** a primary "New task" button (opens Screen 3). A secondary "Agents"
  nav item (Screen 2).
- **States (each must be designed, not assumed):**
  - *No agents yet:* empty board with a single call to action → "Add your first agent"
    (routes to Screen 2). Don't show an empty four-column grid with no path forward.
  - *Agents but no tasks:* columns visible, empty, with "Create a task" prompt.
  - *Populated:* normal board.
  - *Realtime disconnected:* a small "reconnecting…" indicator; on reconnect, refetch the
    board snapshot (don't trust that no events were missed).
  - *Many tasks:* board reads are capped (see read limits); show a count / "showing most
    recent N" rather than silently truncating.

### Screen 2 — Agents (onboarding + key lifecycle, the trust-critical screen)

- **List:** each agent row → name, key-prefix, `last_seen_at` (relative), a **connected
  indicator** (a dot that is grey "not yet connected" until the agent's key is first
  used, then green "connected · last seen 3s ago"), active/revoked badge, and a revoke
  action. The connected state is driven by `last_seen_at` (written on every
  authenticated MCP call) — it confirms a freshly-created key actually works before the
  manager assigns real work. ("Connected" = has made ≥1 MCP call; full board-level
  liveness coloring stays deferred.)
- **Create agent:** name + optional description → on submit, the server generates the API
  key, stores only its SHA-256, and returns the **full key exactly once**.
- **The "shown once" moment (handle carefully — this is where trust is built or broken):**
  - Present the full key in a copy-to-clipboard field with an explicit, unmissable warning:
    "Copy this now — it won't be shown again."
  - Alongside it, show the **ready-to-paste MCP config snippet** (endpoint URL + this key
    in the auth header) so onboarding is copy-paste, not assembly.
  - Require an explicit "I've saved it" confirmation to dismiss; dismissing is
    irreversible (only the prefix remains).
  - If the user loses the key: no recovery — revoke and reissue. State this plainly.
- **Revoke:** confirm (destructive), then `revoked_at` is set; the agent's next MCP call
  fails `401`. Show revoked agents greyed, not deleted (history references them).
- **States:** no agents (the create CTA the board points to); key-just-created (the
  shown-once panel); normal list; revoked rows.

### Screen 3 — Create / assign task

- **Form:** title (required), description (optional), **assign to agent** (required —
  directed assignment, D11; a single-select of the workspace's active agents). Task is
  created as `todo`, assigned, and the board updates live.
- **Edge:** if there are no active agents, the assignee select is empty → block submit and
  route to "Add an agent first" (you can't assign work to nobody).
- **Out of scope for v1 (don't build):** editing/reassigning a task, bulk create,
  priorities/labels/due dates, multi-agent assignment. Create + assign only.

### Visual system (named to prevent AI-slop defaults)

Classifier: **APP UI** (calm, data-dense workspace), so app-UI rules apply.

- **Typography:** a real sans for UI (NOT Inter / Roboto / system-ui — pick a distinctive
  typeface, e.g. a grotesque like Söhne/Geist/IBM Plex Sans and commit) + a **monospace
  for all machine identifiers**: agent ids, key-prefixes, timestamps, task ids, result
  excerpts. Monospace carries meaning here, not decoration.
- **Surface:** calm near-neutral (implementer picks light OR dark and commits to one for
  v1). Minimal chrome, no decorative gradients, no blobs/dividers.
- **Color = status signal, not decoration.** ONE reserved palette: todo neutral,
  in_progress a single calm accent, done muted/green-ish, **failed red** (the only loud
  color, and only when failures exist). Define as CSS variables. No purple/indigo, no
  blue→purple gradients.
- **Cards earn their existence:** task cards are the actual interaction, so cards are
  justified — but they are calm rows, not a bubbly uniform-radius grid with icon-circles.
  No `border-left: 3px accent` cliché; status shown via the column + a small status token.
- **Copy is utility language** (orientation/status/action), not mood/marketing. Section
  headings state what the area is ("Failed", "Agents"), no "Welcome to AgentBoard" hero.
- **Explicitly avoid the AI-slop blacklist:** the 3-column icon-circle feature grid,
  centered-everything, emoji-as-design, decorative shadows, system-ui as the display font.

- **Write-action feedback (create task / create agent / revoke):** pending pattern —

### Responsive & accessibility

- **Responsive (desktop-first, don't-break-on-small):** design for laptop/monitor (the
  manager's real screen). The four columns get min-widths and the board scrolls
  horizontally below tablet rather than crashing; usable down to tablet. A true
  phone-reflow layout (columns → single scrollable stack) is a deferred follow-up.
- **Accessibility (non-negotiable baseline, all viewports):**
  - **Live-region for board updates:** when a card moves columns via Realtime, announce
    it to screen readers via an `aria-live="polite"` region (e.g. "Task X moved to
    Failed"). A silent live board is invisible to SR users — this is the key a11y trap.
  - **Contrast ≥ 4.5:1** on all body/label text — explicitly verify the "calm muted"
    Done column and status tokens, which is exactly where muted palettes fail.
  - **Keyboard:** all actions (New task, open card drawer, create/revoke agent, copy key)
    reachable and operable by keyboard; visible focus rings; drawer traps focus and
    closes on Esc.
  - **Touch targets ≥ 44px** for buttons/interactive rows.
  - **Semantic structure:** columns as labeled regions/landmarks; the copy-key control is
    a real button with an accessible label.

### UI-level decisions worth a design review's attention

- The board is read-only for status (agents drive status via MCP); the human only
  *creates and assigns*. Make that mental model obvious so a manager doesn't hunt for a
  drag-to-move-column interaction that doesn't exist in v1.
- The shown-once key panel is the single highest-stakes UI moment — it deserves the most
  design care of anything here.

## Must-have tests (full coverage of the loop — Eng-review decision 5A)

**Stack:** Vitest for units/integration, Playwright for E2E. 100% coverage of the paths
below is the v1 target — the code is security- and correctness-critical and AI makes the
marginal cost minutes, so tests ship alongside the feature, not as a follow-up.

**Security / isolation (highest stakes):**
- **Cross-tenant isolation (CRITICAL):** two workspaces each owning an agent — assert
  agent A's key, through the full MCP path, can never read or write workspace B's tasks.
  In v1 runs against the app-code scoped query in `lib/agent-db.ts`; keeping it green
  means the scoping predicate can't silently go missing, and it's the regression guard for
  the later RLS retrofit. A failure here is a data breach.
- **Human-plane RLS:** user A's session cannot read user B's workspace rows (verifies the
  `auth.uid()` policies actually deny, not just that the happy path works).
- **Revoked key:** setting `revoked_at` makes the next MCP call `401` immediately.
- **API key:** SHA-256 hashing + prefix; key shown exactly once; lookup by hash matches.

**Core logic (unit):**
- **`lib/task-status.ts`:** full transition matrix — every legal move passes,

## Known risks carried into implementation (not blocking, eyes-open)

- **Remote MCP consumability (outside-voice #14):** the onboarding step "wire the key
  into your agent" assumes the user's agent framework can consume a **remote,
  bearer-authed Streamable-HTTP MCP server** (possibly needing a local proxy for some
  clients). This is the real adoption cliff and the hardest user step. The MCP spike (1A)
  should also confirm at least one *target* client (e.g. Claude Desktop / Cursor / a
  CrewAI or LangGraph agent) can connect to a remote bearer-authed server, not just the
  reference SDK client.
- **Stack vs single-tenant v1 (outside-voice #6):** v1 pays Supabase+RLS+service-role+
  Realtime-RLS complexity for a multi-tenant property it defers (D15). Accepted: the CEO
  review locked the stack (D9) for the v2 trajectory and to avoid a later migration; noted
  here so it's a conscious trade, not an oversight.
- **Rate-limiting / runaway-agent (refinement of the deferred item):** the bigger near-term
  risk isn't abuse, it's a *buggy agent in a tight retry loop* hammering the
  bearer-authed endpoint (cost + DB-write pressure) on day one. The `last_seen_at` throttle
  (D10) blunts the write amplification; a basic per-key rate limit is the first follow-up.

## Appendix A — Agent-plane RLS mechanism (deferred to the multi-user milestone, per D15)

Drop this in when a second tenant first becomes possible (multi-user workspaces). It
replaces the v1 app-code scoped query with database-enforced isolation. Carry agent
identity into the database, per MCP request, inside **one explicit transaction on a single
pooled connection**:

1. **Bootstrap lookup (privileged, scoped to one job):** SHA-256 the bearer key and call a
   **`SECURITY DEFINER`** function `auth_agent(token_hash) returns (agent_id,
   workspace_id)` that does the unscoped `agents` lookup where `revoked_at IS NULL`. This
   resolves the bootstrap paradox: the route handler's own DB role has **no** direct SELECT
   on `agents`, but the definer function (owned by a privileged role) performs that one
   lookup and returns only the two ids. No service-role connection anywhere.
2. **Set identity:** `select set_config('request.agent_id', <id>, true)` and
   `request.workspace_id` (`true` = transaction-local).
3. **Task I/O:** run all task/event queries on the **same transaction**. Agent-plane RLS
   policies read the GUCs: `workspace_id = current_setting('request.workspace_id')::uuid
   AND assigned_agent_id = current_setting('request.agent_id')::uuid`.

**Pooling requirement (make-or-break):** the GUC is transaction-local, so the app **must**
connect through Supabase's pooler (Supavisor/PgBouncer) in **transaction mode**, with
`set_config` + all queries sharing one explicit transaction on one checked-out connection.
Session-mode pooling or out-of-transaction queries would let GUCs leak between requests — a
**silent cross-tenant data leak**. Covered by the cross-tenant test, now re-pointed at the
RLS path.

**Infra to provision:** (a) `CREATE ROLE` for the route handler's low-privilege role (no
direct SELECT on `agents`) and the SECURITY DEFINER owner; wire a separate transaction-mode
`DATABASE_URL`. (b) Enforce the transaction with **`postgres.js` `sql.begin(...)`** (or
equivalent), never supabase-js/PostgREST autocommit. Add the GUC-no-bleed test (two
sequential requests on a reused pooled connection must not share identity).
